# 04 · UX 设计 — molesignal-website

> 产研流水线第四棒（UX）。基于 PRD（03）+ 架构现状（06）+ 现有文案（messages/en.json）进行体验设计。
> 设计立场：改版以提升清晰度与转化为目标，不为改而改。覆盖 PRD 全部功能域 D1–D9。
> 设计时间：2026-06-01。

---

## 1. 核心用户旅程

### 旅程 A：访客 → 理解价值 → 上手 → 转化（主漏斗，7步）

这是官网存在的核心路径，每一步都有明确的用户目标和系统响应。

| 步骤 | 用户位置 | 用户目标 | 系统响应 | 关键设计决策 |
|---|---|---|---|---|
| **A-1** 落地 | `/`（首页 Hero）| "这是什么，和 Datadog 有什么不同？" | 展示定位句 + pre-1.0 标签 + GitHub star chip（建立初步信任） | Hero 主标题二行结构：定位 + 省钱hook；不放多个 CTA，**主 CTA 只有一个** |
| **A-2** 摸产品手感 | `/`（CrossSignalDemo 区块）| "它真的能做跨信号关联？让我看看" | 自动轮播 trace/logs/metric 三 tab，keyboard可操作；同一 trace_id 贯穿三视图 | 此处不放 CTA，让用户专注体验；只在末尾放"How does this work →"指向架构页 |
| **A-3** 理解省钱 | `/why`（成本计算器）| "真能省多少？给我数字" | 双滑块 CostCalculator 即时计算；CompareTable 7维对比；5个真实场景 | `/why` 是**转化钩子的核心页**，计算器拖动是关键埋点事件；成功态应展示"你大概能省 $X/月"大字 |
| **A-4** 验证技术可行性 | `/architecture` | "架构是否成熟？我能驾驭吗？" | 交互式数据路径图 + 8段 sticky TOC + ingest 表 | sticky TOC 让工程师快速跳到关心的段（query engine / multi-tenancy），不强迫从头读 |
| **A-5** 动手安装 | `/start` | "5 分钟能跑起来吗？" | docker/helm/binary 三 tab + 可一键复制的命令 + OTLP/SQL 示例 | helm/binary 命令加"v1.0 target"标签诚实说明现状，docker tab 是主推路径 |
| **A-6** 转化分叉 | `/cloud` 或 `/design-partner` | 分叉1："不想自运维，留邮箱等托管" / 分叉2："想深度共创，申请伙伴" | 候补表单（单邮箱，低门槛）/ 设计伙伴申请（5字段，高参与度）| 两个路径在首页 + `/start` 末尾 + Pricing 页都有入口；主路径CTA不互相干扰 |
| **A-7** GitHub star | 任意页的 GitHub chip | "认可项目，想关注进展" | chip 打开 GitHub 仓库页；star 数量本身是信任信号 | chip 不应在每个页面"抢镜"，在 Hero 区 + `/architecture` 末尾 + blog 文章页出现即可 |

**埋点触发点（P0-5）**：
- A-2 轮播 tab 切换 → `demo_tab_switch`
- A-3 CostCalculator 首次拖动滑块 → `cost_calculator_interact`
- A-3 CompareTable 展开"See full comparison" → `compare_table_expand`
- A-5 CodeBlock 复制命令 → `quickstart_copy`（携带 tab 参数：docker/helm/binary）
- A-6 候补表单提交成功 → `waitlist_submit`
- A-6 设计伙伴表单提交成功 → `design_partner_submit`
- 任意 CTA 点击 → `cta_click`（携带 label 与 source_page 参数）
- A-7 GitHub chip 点击 → `github_star_click`

### 旅程 B：开源关注者回访（内容消费，3 条支线）

| 支线 | 路径 | 用户目标 | 关键设计 |
|---|---|---|---|
| **B-1 路线图** | `/roadmap` | "看看下一步做什么，我的需求有没有" | Now/Next/Later/Done 四 tab，URL hash 同步（已实现）；每条路线图项链接 GitHub issue（Proof over promise） |
| **B-2 更新日志** | `/changelog` | "上周发了什么？项目还活着吗？" | 真实 GitHub Releases（ISR）> 静态回退；侧栏版本锚快速跳转；RSS 入口醒目；M2 前用"previewing"标签诚实标注非真实 release |
| **B-3 博客** | `/blog` → `/blog/[slug]` | "看深度技术文章，理解设计决策" | EN-only（设计有意）；ZH 页面给友好提示 + 跳 EN 链接；相关文章按 tag 聚合；M2 前 blog body 是 Markdown-lite 纯文本，M2 接线 MDX 后升级为富文本 |

### 旅程 C：多语言访问（EN ↔ ZH）

| 步骤 | 场景 | 关键设计 |
|---|---|---|
| **C-1** 切换语言 | 任意页头部 LocaleSwitcher | 保留滚动位置；EN 在根 `/`，ZH 在 `/zh/*`；切换时触发 `locale_switch` 埋点 |
| **C-2** ZH 访问 Blog | `/zh/blog*` | 显示"Blog is English-only in v1"友好提示 + 明显的"Read in English →"链接；不是 404，是体验降级而非中断 |
| **C-3** i18n 缺口 | 任何 ZH 键缺失 | 回退到 EN 文本（不留空白、不报错）；脚本门禁（lint:i18n）保证键结构对齐 |

---

### 旅程 D：辅助流程

- **主题切换**：Nav 右侧图标，明/暗/系统；localStorage 持久化，根 layout 内联脚本防闪烁
- **PreReleaseBanner 关闭**：顶部横幅可关闭，7 天后重现；不阻碍核心浏览
- **移动端导航**：TopNav 汉堡→Sheet 抽屉，主导航+语言+主题控件均在抽屉内
- **SkipToContent / ScrollToTop**：a11y 标配，键盘导航入口

### 2.2 全局导航结构

#### TopNav（桌面端，从左到右）

```
[molesignal logo]  Why  Architecture  Quick Start  Pricing  Cloud  [Resources▼]  |  [GitHub chip]  [主题]  [语言]
```

**Resources 下拉**（仅桌面）：
```
Resources ▼
├── Roadmap        "What's coming"
├── Changelog      "Release notes"
├── Blog           "Engineering posts"
└── Download       "Releases & binaries"  ← 诚实标注"coming soon / v1.0 target"
```

> **改版调整**：现状 Nav 有 Docs 入口（指向不存在的 docs.molesignal.io）。**移除 Docs 入口，或改为 Resources 下拉 → Quick Start（站内文档承载）**。Quick Start 既作为顶级 Nav 项（主路径），也是 Resources 下"文档"的替代。"Download" 保留在 Resources 下但加 coming-soon 状态（不可点，或点击展示 tooltip）。

#### TopNav（移动端）

```
[molesignal logo]  [hamburger]
→ Sheet 抽屉:
   Why / Architecture / Quick Start / Pricing / Cloud
   ── Resources ──
   Roadmap / Changelog / Blog / Download(coming soon)
   ── ──
   [GitHub chip]  [主题]  [语言]
```

#### Footer 结构

```
[Product列]        [Resources列]      [Community列]    [Legal列]
Why molesignal     Roadmap            GitHub           Privacy
Architecture       Changelog          Discord           Terms
Quick Start        Blog               Twitter / X
Pricing            Download
Cloud
Design Partner
```

**Footer 改版调整（P0-4 修复死链）**：
- `Docs` 链接 → **移除**，不指向不存在的 `docs.molesignal.io`
- `Discord` → 改为禁用态（灰色文字 + "launching soon" tooltip），不是 `href="#"`
- `Download` → 加 "v1.0 target" 文案，不可点或点击跳 `/start` 并锚定 binary tab
- 保留 `GitHub` 链接（真实 URL）

### 2.3 改版调整汇总（相对现状）

| 改动点 | 改前 | 改后 | 理由 |
|---|---|---|---|
| Nav Docs 入口 | 指向 `docs.molesignal.io`（死链） | 移除或替换为 Quick Start | Proof over promise；消灭死链（P0-4） |
| Footer Discord | `href="#"`（占位） | 禁用态 + "launching soon" | 同上 |
| Footer Download | 指向未就绪产物 | coming-soon 标注或禁至 M2 | 同上 |
| 首页区块顺序 | Hero → Demo → Why → Arch → Start → Stats → DP | 调整为 Hero → Demo → Why/Stats → Arch → Start → DP | 让"证明活跃"（Stats）紧跟"为什么"强化信任，减少进入 QuickStart 前的犹豫 |
| `/start` Helm/Binary tab | 命令可复制但可能不能跑 | 加 "v1.0 target" pill 标注 | 诚实文案（P0-4） |
| Cloud 候补成功态 | 仅 toast | 页面内持久成功态 + 下一步引导 | 减少提交后的迷失感 |
| Design Partner 成功态 | 仅 toast | 持久成功态卡片 + "回复时间 48h" 期望管理 | 同上 |
| Privacy/Terms | 占位文本 | 真实法务文本 | 合规（P0-6） |

## 4. 状态设计

### 4.1 表单状态（Cloud 候补 + Design Partner 通用）

| 状态 | 触发条件 | UI 表现 | 文案要点 |
|---|---|---|---|
| **默认空态** | 页面加载 | 表单字段为空，Button 可点 | label + placeholder 清晰 |
| **输入中** | 用户填写字段 | 实时校验（onBlur），错误立即显示 | 不打扰式：只在失焦时校验，不在打字时报错 |
| **校验错误** | 点 Submit 或 onBlur | 字段 border 红色 + 行内错误文字（字段下方） | 具体说明错误："Please enter a valid email"；不说"Field required"这种模糊话 |
| **提交中** | 点 Submit + 校验通过 | Button 变 disabled + spinner + "Submitting…" | 防止用户多次点击；字段保持只读 |
| **成功** | API 返回 2xx | 原位替换 → 成功卡片（持久，不 dismiss） | 确认用户动作 + 下一步建议（不让用户迷失） |
| **服务端错误** | API 返回 4xx/5xx（非限流） | toast 错误（sonner）+ 建议直接邮件联系 | "Submission failed — please retry, or email founders@molesignal.io directly." |
| **限流命中** | API 返回 429 | toast 温和提示（非红色报错风格） | "Too many attempts. Try again in a few minutes." |
| **网络断开** | fetch 失败 | toast 错误 + 提示检查网络 | 与 API 错误区分文案 |

### 4.2 内容空态

| 场景 | 空态触发 | UI 表现 | 文案要点 |
|---|---|---|---|
| **Changelog 无 release** | GitHub Releases 为空且静态回退也无数据 | 居中空态卡：图标 + 标题 + 正文 + CTA | "No releases yet. Watch the repo for notifications, or grab the RSS feed." |
| **Blog 无文章** | `content/blog.ts` 为空 | 空态卡：图标 + "We're writing the first launch article." + 副CTA | 不展示空列表，友好引导 |
| **Roadmap 某 Tab 无条目** | 该 tab 数组为空 | "Nothing in this phase yet. Check the others." 小文字居中 | 不报错，不让用户以为加载失败 |
| **贡献者墙无数据** | GitHub API 失败/私有仓库 | 只显示 "Be the first contributor → open an issue or PR"（现有文案）| 不显示空头像格，不造成"项目无人"的误读 |

### 4.3 GitHub 数据兜底态

| 数据 | 正常态 | 兜底态 | 原则 |
|---|---|---|---|
| **Stars / 最近提交** | 真实数字 + chip | "★ on GitHub"（无数字） + chip | 不伪造数字；chip 本身是可点的，引导到真实 GitHub 页 |
| **贡献者墙** | 真实头像 + 链接 | 空态 + "Be the first contributor" | 私有/失败不报错，不展示假头像 |
| **Changelog** | 真实 releases | 3条静态回退 + "previewing" pill | 标注来源，不假装是真实 release |

### 4.4 i18n 缺失兜底

| 场景 | 处理方式 |
|---|---|
| ZH 键缺失 | 回退 EN 文本，不显示键名 |
| `/zh/blog*` | 显示 EN-only 提示横幅 + "Read in English →"，不是 404 |
| ZH 法务页 | 显示 EN 文本（P0-6 要求 EN/ZH 双语，M1 前若只有 EN，ZH 回退 EN） |
| 日期/数字格式 | 使用 `Intl` 按 locale 格式化，不硬编码 |

```

Button disabled + spinner
    ↓
```

## 6. 给 UI 设计师的提示

### 6.1 情绪基调（产品调性）

**关键词**：工程师向 / 诚实硬核 / Proof over promise / 反浮夸

- **不要**：渐变大横幅配空洞文案、无数据的"指标墙"（"10x faster" 无依据）、企业营销感
- **要**：数字真实可追溯（成本计算器公式可验证）、截图展示真实产品 UI 而非美化样机、Changelog 的 diff 风格
- 参考气质：Linear、Fly.io、Litestream 的官网 —— 极简、文字驱动、代码高亮、暗色主题友好

### 6.2 重点视觉表达区块

| 区块 | 视觉侧重点 | 说明 |
|---|---|---|
| **Hero** | 标题排版张力 | 两行 H1 的对比感（"What you'd build" vs "Without the Datadog bill"）是核心传达；用渐变或 highlight 色强调第二行 |
| **CrossSignalDemo** | 终端/代码审美 | tab 切换区域类似 IDE 标签；背景暗色代码区；3 个信号间的关联感（同一 trace_id 高亮） |
| **CostCalculator** | 大数字冲击 | "Savings" 卡的金额数字要大；动态变化时有过渡动效；绿色代表节省，红色代表 Datadog 成本 |
| **Architecture Diagram** | 技术精确感 | 流程图节点风格干净、技术感强；tooltip hover 交互反馈清晰；不要卡通化 |
| **表单成功态** | 正向反馈感 | ✓ 图标 + 绿色调；不要过于 celebratory（这是工程师产品，不是 SaaS 促销网站） |
| **PreReleaseBanner** | 轻量信息条 | 诚实"pre-1.0"状态；不遮挡内容，可关闭；不要警告红色，用中性色 |

### 6.3 组件样式参考

- **现有 token**：4层 OKLCH 级联（亮色/暗色），品牌色 Indigo + 营销粉 `chart-5`，在 `app/globals.css`
- **组件库**：shadcn/ui（`radix-nova` style，new-york 衍生），不自造轮子
- **代码高亮**：Shiki SSR 双主题（亮/暗色对应 globals.css token），已有实现，保持一致
- **字体**：Inter Variable 自托管，不替换
- **动效**：framer-motion，已有 `useReducedMotion` 守卫，保持；动效宜克制（工程师受众不喜欢过度动画）

### 6.4 移动端优先级

- TopNav 折叠为汉堡 + Sheet 抽屉，抽屉内容见 §2.2
- CostCalculator 双滑块在移动端需足够大的触摸区域（44px 最小）
- CrossSignalDemo 三 tab 在小屏下 tab 文字可缩短（Trace/Logs/Metric 保持，不截断）
- CompareTable 在移动端可水平滚动，不裁切列
