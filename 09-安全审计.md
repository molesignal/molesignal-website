## 2. 风险清单

### 🔴 高危（建议本工单内修复，至少须经主管/PM 显式接受残留风险）

#### H1 — 限流主键可被 `X-Forwarded-For` 伪造绕过（限流形同虚设）
- **位置**：`lib/rate-limit.ts:59-65`，核心在 `:61` `return xff.split(",")[0].trim();`
- **类型**：CWE-290 通过伪造可信数据绕过限流 / CWE-307 不当的请求频率限制（OWASP API4:2023 Unrestricted Resource Consumption）
- **严重级别**：**高**（在 ISSUE-1 的"可靠防刷"目标语境下）
- **利用场景**：限流键为 `{form}:{ip}`，IP 取 `x-forwarded-for` 的**最左段**。该段是**客户端可任意伪造**的请求头——攻击者每次请求带不同的 `X-Forwarded-For: <随机IP>`，即可让每次请求都落到全新的限流桶，**完全绕过 10/h、5/h 阈值**，对候补/DP 表单无限灌水、刷爆创始人收件箱、污染 Resend 名单。这恰好是本工单（P0-3"防刷实际失效"）要解决的问题——**若只修跨实例共享而不修取信源，面对蓄意攻击限流依然失效**，Upstash 改造的安全收益被架空。
  - Vercel 平台下：客户端发送的 `X-Forwarded-For` 会被 Vercel 代理**追加**真实 IP（真实 IP 在右侧/末段），最左段保留攻击者伪造值；而 `x-real-ip` 由 Vercel 边缘设置、客户端不可覆盖，是更可信的取信源。当前代码恰恰**优先用了可伪造的 XFF 最左段**，仅在 XFF 缺失时才用 `x-real-ip`（顺序反了）。
- **修复建议**（改动极小，单文件一处）：
  1. **优先信任平台注入的不可伪造来源**。Vercel 上优先 `x-real-ip`；或取 `x-forwarded-for` 的**最后一段**（Vercel 追加的真实 IP），而非第一段：
     ```ts
     export function getClientIp(req: Request): string {
       // Vercel 边缘设置、客户端不可覆盖，最可信
       const real = req.headers.get("x-real-ip");
       if (real) return real.trim();
       // 回退：XFF 取「最后一段」（反向代理追加的真实连接 IP），而非可伪造的首段
       const xff = req.headers.get("x-forwarded-for");
       if (xff) {
         const parts = xff.split(",").map((s) => s.trim()).filter(Boolean);
         if (parts.length) return parts[parts.length - 1];
       }
       return "unknown";
     }
     ```
     > 注：取信策略取决于实际部署拓扑（信任的代理跳数）。Vercel 单层代理下「`x-real-ip` 优先」即可；若未来加自有 CDN/WAF，需按"信任边界后第 N 段"取，并文档化信任的代理数。
  2. 若本工单不修（沿用现状），**必须**由主管/PM 显式记录"限流可被 header 伪造绕过"为本期已接受残留风险，并将"取信源加固"登记为紧后置工单——否则 ISSUE-1 的 AC（可靠限流）在威胁模型下不成立。

> **诚实定性**：本项虽利用门槛极低（一行脚本），但现实影响**受限**（名单灌水 + 收件箱噪声，无机密泄露/无资金损失），且本期已显式排除验证码（工单"明确不做"）。判**高**的理由是它**直接架空本工单要建立的 P0 安全控制**且修复成本极低；不判"发布阻断级"是因影响面有限。建议处置：**本工单顺手修**（性价比最高）。

### 🟡 中危

#### M1 — 内存兜底 Map 无淘汰/清扫，配合 H1 构成内存耗尽 DoS 面
- **位置**：`lib/rate-limit.ts:17`（`const STORE = new Map()`）、`:34-52`（仅在同 key 再次命中时覆盖，过期桶从不主动删除）
- **类型**：CWE-400 不受控的资源消耗（内存）
- **严重级别**：**中**（仅影响内存兜底路径；受 serverless 实例短生命周期约束，但单实例生命周期内仍可被打爆）
- **利用场景**：`STORE` 以 IP 为键、**无上限、无定期清扫**，过期条目只有在"同一 key 再次到达"时才被覆盖回收。结合 H1（伪造 XFF 注入无限唯一键），攻击者可在单个 serverless 实例存活期内向 `STORE` 注入海量唯一键 → 内存线性膨胀 → 函数 OOM/降速。本期"不提供 UPSTASH 密钥、走内存兜底"的决策使该路径成为**默认生产路径**，风险被放大。
- **修复建议**：在 `memoryRateLimit` 中加**容量上限 + 惰性清扫**：每次写入前若 `STORE.size` 超阈值（如 10k），清除一批已过期条目；仍超限则对新键 fail-open（放行 + 一次 warn），避免无界增长。示例：
  ```ts
  const MAX_KEYS = 10_000;
  function sweepIfNeeded(now: number) {
    if (STORE.size < MAX_KEYS) return;
    for (const [k, b] of STORE) if (b.resetAt <= now) STORE.delete(k);
  }
  ```
- **关联**：修复 H1（取信不可伪造来源）后，唯一键基数回落到真实 IP 量级，本项风险大幅下降；二者建议一起处理。

### 🟢 低危 / 配置与合规

#### L1 — `Retry-After` 可能为负数（健壮性，AC4 已要求修）
- **位置**：`app/api/cloud-waitlist/route.ts:26`、`app/api/design-partner/route.ts:28`：`String(Math.ceil((rl.resetAt - Date.now()) / 1000))`
- **类型**：HTTP 头取值健壮性（非安全直接问题）
- **级别**：低
- **场景**：时钟漂移/竞态下 `resetAt < now` 会算出负秒数，部分客户端对负 `Retry-After` 行为不确定。
- **修复**：按工单 AC4 / 架构 §4.3 加非负保护 `Math.max(0, Math.ceil((rl.resetAt - Date.now()) / 1000))`。**落地实现须确保这两处都改到**（当前两处均缺保护）。

#### L2 — IP / Email 等 PII 进日志与通知邮件正文（隐私合规）
- **位置**：`cloud-waitlist/route.ts:49`（`console.info` 记 email）、`:53`（邮件正文含 `IP`）；`design-partner/route.ts:65,71`（同上）
- **类型**：CWE-532 敏感信息写入日志 / GDPR（IP 属个人数据）
- **级别**：低（站点性质 + 数据量小；但需合规背书）
- **场景**：IP 与邮箱写入 Vercel 函数日志与发往创始人邮箱的正文，留存于第三方（Vercel/Resend）。须由隐私政策（P0-6）覆盖"收集 IP/邮箱、用途、留存、第三方处理者(Resend/Vercel/Plausible)"。
- **修复建议**：① 确认隐私政策（T-法务）明确披露 IP 收集与限流用途；② 日志中可对 IP 做截断/哈希（如保留 /24 或 `sha256(ip+salt)` 前 8 位）以最小化；邮件正文留 IP 可接受（运营反滥用取证需要），但应在隐私政策标注。**本项不阻塞本工单**，转隐私政策工单跟进。

#### L3 — Upstash REST Token 为全量 Redis 凭据（密钥卫生）
- **位置**：`.env.local.example:16-17`（`UPSTASH_REDIS_REST_URL/TOKEN`，占位符）；落地后用于 `Redis.fromEnv()`/手动构造
- **类型**：密钥最小权限 / 泄露面
- **级别**：低（当前配置正确，前瞻提醒）
- **风险**：Upstash REST token 通常具完整读写/`FLUSHDB` 权限；一旦泄露（如误加 `NEXT_PUBLIC_` 前缀进客户端包、误打日志）攻击者可读写/清空限流数据。
- **修复建议（落地时核对）**：① 两个 `UPSTASH_*` **严禁** `NEXT_PUBLIC_` 前缀（仅服务端 route handler 读取，确保不进客户端 bundle）；② 不在任何日志打印 token/url；③ 若 Upstash 控制台支持，使用**只读/限定命令**的 token 或独立 DB 隔离；④ 制定泄露后轮换流程。**当前 `.env.example` 与 `.gitignore` 配置正确，无需改动**。

#### L4 — 依赖：postcss < 8.5.10 中危 XSS（传递依赖，构建期）
- **位置**：`pnpm audit --prod` 结果：`postcss <8.5.10`（经 `next` / `next-intl>next` 传递），GHSA-qx2v-qp2m-jg93
- **类型**：依赖已知漏洞（CWE-79，构建期 CSS 处理）
- **级别**：低（**仅构建期**处理项目自有 CSS，无用户提供的不受信 CSS 输入 → 现实可利用性极低）
- **修复建议**：随手处理——`package.json` 加 `pnpm.overrides`：`"postcss": ">=8.5.10"`，或升级 `next` 至已修版本后 `pnpm install`；非本工单阻塞项。

#### L5 — 缺 CSP 与 HSTS 头（纵深防御，非本工单）
- **位置**：`vercel.json:6-19` —— 已配 `X-Content-Type-Options`/`X-Frame-Options`/`Referrer-Policy`/`Permissions-Policy`（✅ 良好基线），但**缺 `Content-Security-Policy` 与 `Strict-Transport-Security`**
- **级别**：低（与限流工单无关，记录供通审/devops 跟进）
- **修复建议**：上线前补 `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`；CSP 因含内联脚本/Plausible 需配 nonce/allowlist，单独工单处理。

#### 信息性 / 理论项（无需动作）
- **CSRF**：两个写接口为公开无 Cookie/Session 鉴权的 POST，不存在"借受害者凭据触发状态变更"的前提 → **CSRF 不适用**。蜜罐是反机器人手段，非 CSRF 防护。无需处理。
- **验证码/人机校验**：工单"明确不做"已显式排除（超本期范围）。在 H1 修复 + honeypot 之上，本期反滥用为"尽力而为"，可接受。
- **共享出口 IP（NAT/企业代理）**：纯 per-IP 限流会使同出口的多个真实用户被合并计数（可用性折损），对候补表单可接受，记录即可。

- **位置/证据**：`lib/email.ts`（仅 `sendEmail()` 调 `https://api.resend.com/emails` 发一封通知邮件，**无任何 contacts/Audiences API**）；`app/api/cloud-waitlist/route.ts:50-56`、`app/api/design-partner/route.ts:72-82`（成功路径仅 `sendEmail({to: foundersEmail()})`，未入任何名单）；`app/api/cloud-waitlist/route.ts:11-13` 注释明言「(eventually) adds to a Resend audience… **For now it's just the founders notification**」；`.env.local.example` 的 `RESEND_CLOUD_AUDIENCE_ID`/`RESEND_PARTNER_AUDIENCE_ID` **未被任何代码引用**（计划态，未实现）。
- **类型/级别**：AC5 事实不符（过度声称）/ **高**（合规红线核心）。
- **后果**：若 fullstack 照工单"真实数据流 C"原话写「Resend 存储你的邮箱/姓名于候补名单(Audiences)」，即声称一个**当前不存在**的数据存储行为——恰违反 AC5「不声称站点不存在的行为」。
- **✅ 正确口径（照此写）**：「我们使用 **Resend** 作为邮件服务商，把含你提交内容的**通知邮件**投递给创始团队。Resend 作为邮件处理方（子处理者）会处理这封邮件的传输。**我们目前不会把你加入任何 Resend 营销名单 / Audiences。**」（若日后启用 Audiences 入库，须同步更新本页——可在文案留"如未来变更将更新本政策"。）

#### F2 — IP 表述不完整：IP（及邮箱/姓名）会随通知邮件正文发给创始团队并长期留存，不止"限流瞬态"
- **位置/证据**：`app/api/cloud-waitlist/route.ts:54` `text: \`${email}\n…\nIP: ${ip}\n…\`` ；`app/api/design-partner/route.ts:66` `\`IP: ${ip}\``。即每次提交，**提交者 IP + 时间戳 + 全部表单字段**被写入通知邮件正文，经 **Resend 传输**、长期留存于**创始人邮箱**。（本仓既有审计 §2 **L2** 已记录此 PII 流向，ISSUE-4 正是它必须被披露的时点。）
- **类型/级别**：AC5 漏报真实数据共享/留存 / **高**。
- **后果**：若隐私页照工单 B 段口径只写「IP 仅用于防滥用限流、短期/瞬态、不做画像」，则**漏报** IP 实际还被邮件化、经第三方(Resend)传输并留存——低估数据共享与留存期，AC5 不实。
- **✅ 正确口径（补这句）**：「当你提交表单时，为便于我们跟进，你的**提交内容连同提交时的 IP 与时间戳**会包含在发给创始团队的通知邮件中（经 Resend 传输并保留在我们的邮箱中）。**IP 另用于**接口防滥用限流（按 IP 计数、短期）。」（其余"不做画像/不出售"属实，可保留。）
