## 2. 风险清单

### 🔴 高危（发布前必修）

**本期设计未发现已落地的高危漏洞。** `addContact` provider 的注入/SSRF/密钥面已在 ISSUE-8 审计清零（`audienceId` 经 `encodeURIComponent`、key/id 仅来自 env 不可被请求体覆盖、错误日志截断 body 不打印 key——见 `lib/resend-audiences.ts:56/40-52/85`）。以下为 **route 接线开发时若做错即升为高危** 的强约束，列为「开发期不可逾越红线」（QA 验 AC5 时逐条确认）：

- **G1（接线顺序——蜜罐必须前置于一切副作用）· 命中即高危**
  - 现状 `route.ts:51-53` 蜜罐判定（`website` 非空 → silent `200`）在 `sendEmail` **之前**，正确。接线 `addContact` 时**必须**把 `Promise.allSettled([addContact, sendEmail])` 并行块放在蜜罐 `return` **之后**（即 `route.ts:53` 之后），与 cloud-waitlist 模板 (`app/api/cloud-waitlist/route.ts:50-70`) 完全一致。
  - **若做错**（addContact 写在蜜罐判定之前 / 或蜜罐改成"填了也继续"）→ bot 提交会被写入**可营销 partner audience** 并触发通知邮件，污染 NSM 合格转化数 + 给创始邮箱灌垃圾。属 AC5③ 红线。**QA 必验**：蜜罐命中 → 零 contacts 请求、零通知、零 track（参照测试方案"安全/不回归"分支）。
- **G2（入库字段最小化——只传 email + first/last）· 命中即高危**
  - `addContact` 调用**仅可传** `audienceId`(env) / `email` / `firstName` / `lastName` / `unsubscribed?`。**严禁**把 `companySize` / `currentStack` / `biggestPain` / `ip` 作为 contact 字段塞入（provider 签名也不接收，但禁止为塞这些而改签名）。这是 AC5① 的硬约束——结构化字段只走通知邮件正文归档，不进可营销名单（最小化 PII 外泄面 + 避免把"痛点自述"这类敏感文本同步给营销侧）。

## 2. 风险清单

### 🔴 高危（建议本工单内修复，至少须经主管/PM 显式接受残留风险）

#### H1 — 限流主键可被 `X-Forwarded-For` 伪造绕过（限流形同虚设）
- **位置**：`lib/rate-limit.ts:59-65`，核心在 `:61` `return xff.split(",")[0].trim();`
- **类型**：CWE-290 通过伪造可信数据绕过限流 / CWE-307 不当的请求频率限制（OWASP API4:2023 Unrestricted Resource Consumption）
- **严重级别**：**高**（在 ISSUE-1 的"可靠防刷"目标语境下）
- **利用场景**：限流键为 `{form}:{ip}`，IP 取 `x-forwarded-for` 的**最左段**。该段是**客户端可任意伪造**的请求头——攻击者每次请求带不同的 `X-Forwarded-For: <随机IP>`，即可让每次请求都落到全新的限流桶，**完全绕过 10/h、5/h 阈值**，对候补/DP 表单无限灌水、刷爆创始人收件箱、污染 Resend 名单。这恰好是本工单（P0-3"防刷实际失效"）要解决的问题——**若只修跨实例共享而不修取信源，面对蓄意攻击限流依然失效**，Upstash 改造的安全收益被架空。
  - Vercel 平台下：客户端发送的 `X-Forwarded-For` 会被 Vercel 代理**追加**真实 IP（真实 IP 在右侧/末段），最左段保留攻击者伪造值；而 `x-real-ip` 由 Vercel 边缘设置、客户端不可覆盖，是更可信的取信源。当前代码恰恰**优先用了可伪造的 XFF 最左段**，仅在 XFF 缺失时才用 `x-real-ip`（顺序反了）。
- **修复建议**（改动极小，单文件一处）：
  1. **优先信任平台注入的不可伪造来源**。Vercel 上优先 `x-real-ip`；或取 `x-forwarded-for` 的**最后一段**（Vercel 追加的真实 IP），而非第一段：
     ```ts
     export function getClientIp(req: Request): string {
       // Vercel 边缘设置、客户端不可覆盖，最可信
       const real = req.headers.get("x-real-ip");
       if (real) return real.trim();
       // 回退：XFF 取「最后一段」（反向代理追加的真实连接 IP），而非可伪造的首段
       const xff = req.headers.get("x-forwarded-for");
       if (xff) {
         const parts = xff.split(",").map((s) => s.trim()).filter(Boolean);
         if (parts.length) return parts[parts.length - 1];
       }
       return "unknown";
     }
     ```
     > 注：取信策略取决于实际部署拓扑（信任的代理跳数）。Vercel 单层代理下「`x-real-ip` 优先」即可；若未来加自有 CDN/WAF，需按"信任边界后第 N 段"取，并文档化信任的代理数。
  2. 若本工单不修（沿用现状），**必须**由主管/PM 显式记录"限流可被 header 伪造绕过"为本期已接受残留风险，并将"取信源加固"登记为紧后置工单——否则 ISSUE-1 的 AC（可靠限流）在威胁模型下不成立。

> **诚实定性**：本项虽利用门槛极低（一行脚本），但现实影响**受限**（名单灌水 + 收件箱噪声，无机密泄露/无资金损失），且本期已显式排除验证码（工单"明确不做"）。判**高**的理由是它**直接架空本工单要建立的 P0 安全控制**且修复成本极低；不判"发布阻断级"是因影响面有限。建议处置：**本工单顺手修**（性价比最高）。

### 🟡 中危

#### M1 — 内存兜底 Map 无淘汰/清扫，配合 H1 构成内存耗尽 DoS 面
- **位置**：`lib/rate-limit.ts:17`（`const STORE = new Map()`）、`:34-52`（仅在同 key 再次命中时覆盖，过期桶从不主动删除）
- **类型**：CWE-400 不受控的资源消耗（内存）
- **严重级别**：**中**（仅影响内存兜底路径；受 serverless 实例短生命周期约束，但单实例生命周期内仍可被打爆）
- **利用场景**：`STORE` 以 IP 为键、**无上限、无定期清扫**，过期条目只有在"同一 key 再次到达"时才被覆盖回收。结合 H1（伪造 XFF 注入无限唯一键），攻击者可在单个 serverless 实例存活期内向 `STORE` 注入海量唯一键 → 内存线性膨胀 → 函数 OOM/降速。本期"不提供 UPSTASH 密钥、走内存兜底"的决策使该路径成为**默认生产路径**，风险被放大。
- **修复建议**：在 `memoryRateLimit` 中加**容量上限 + 惰性清扫**：每次写入前若 `STORE.size` 超阈值（如 10k），清除一批已过期条目；仍超限则对新键 fail-open（放行 + 一次 warn），避免无界增长。示例：
  ```ts
  const MAX_KEYS = 10_000;
  function sweepIfNeeded(now: number) {
    if (STORE.size < MAX_KEYS) return;
    for (const [k, b] of STORE) if (b.resetAt <= now) STORE.delete(k);
  }
  ```
- **关联**：修复 H1（取信不可伪造来源）后，唯一键基数回落到真实 IP 量级，本项风险大幅下降；二者建议一起处理。

## 隐私与访问控制（合规）

> 导出的 CSV 含可营销邮箱与 Design Partner 真实姓名/公司画像，属高敏 PII。处理时遵守以下约定，与站点隐私政策（/privacy）口径一致。

- **Resend Dashboard**：仅创始团队成员，各自独立账号、启用 Resend 团队最小权限，禁共享登录。
