## 1. 一句话现状 + 技术栈实况

**一句话**：molesignal-website 是一个**完成度相当高的纯前端营销官网**（"Datadog 平替"的自托管可观测性开源产品官网），17 条路由、28 个 SSG 页面、EN/ZH 双语全量翻译、设计系统/无障碍/SEO 基建齐备；**唯二真正"有后端的功能"**是两个表单（Design Partner 申请、Cloud 候补邮箱）的 API 路由 + GitHub 数据拉取——但这两块都**依赖外部环境变量（Resend / GitHub Token）才真正生效，本地/未配置时优雅降级为 noop**。其余全部是静态内容 + 客户端交互动效。**没有数据库、没有自有后端服务、没有用户认证、没有真实订阅存储。**

**技术栈实况**（`package.json`）：

| 维度 | 实况 | 备注 |
|---|---|---|

## 5. i18n 现状

- **语言**：EN（默认，根路径）+ ZH（`/zh/*`）。`i18n/routing.ts` `localePrefix: as-needed`，`/en/*` 301 回根。
- **路由机制**：`proxy.ts`（Next16 新名，取代 middleware）调 `next-intl/middleware`；matcher 排除 `api`/`_next`/`_vercel`/静态文件。服务端 `i18n/request.ts` 按 locale 动态 import 消息包。导航用 `i18n/navigation.ts` 的本地化 `<Link>`。
- **文案完整度**：`messages/en.json` 与 `zh.json` **同为 626 行**，由 `scripts/check-i18n-parity.ts` 强制键结构一致（差异即 exit 1）。**双语全量翻译，无明显缺口**。
- **例外**：**Blog（列表+详情）EN-only**（设计有意，见 DESIGN_BRIEF "Out of Scope"）。`/zh/blog*` 渲染友好"不可用"提示并引导回 EN。sitemap 对 blog 不输出 hreflang alternates。

## 7. 关键约束、技术债与风险

1. **"功能"本质是营销外壳**：除两个表单外无任何业务后端。要把"看起来有的功能"变真，最大硬骨头是**从零引入数据持久层**（候补名单、表单存档、可能的内容管理）。
2. **Cloud 候补无存储**（#12）：当前只发通知邮件，**无法批量触达/导出名单**——产品角度这是"假候补"，红线级（涉数据沉淀与后续增长）。
3. **限流不可靠**（#13）：内存 Map 在 Vercel serverless 多实例下形同虚设，防刷能力弱。换 Upstash Redis 才靠谱。
4. **依赖未上线的外部产物**：QuickStart 的 helm repo（`charts.molesignal.io`）、binary release URL、Footer 的 `docs.molesignal.io`、Discord 邀请链接——**全是"画饼"占位**，用户照做会失败/死链（信任风险，与 DESIGN_BRIEF "Proof over promise" 原则直接冲突）。
5. **内容硬编码、MDX 半成品**：blog/changelog/roadmap 全是 TS 数组，更新内容要改代码+发版；MDX 依赖装了未接线，blog body 当纯文本渲染（无富文本/代码块/图）。无 CMS。
6. **changelog/stats 真实性取决于仓库公开度**：若 `molesignal/molesignal` 未公开或无 Release，GitHub 数据全回退到静态/空态，"building in the open" 的卖点落空。
7. **零自动化测试 + 零 CI**：所有质量门禁靠人记得手跑；回归风险高。下游若要迭代功能，应优先补 CI（把 `pnpm check` + build 串进 Actions）和关键路径 E2E（表单提交、i18n、链接）。
8. **法务页占位**（#28）：privacy/terms 仅占位文案，上线前需真实法务文本（尤其涉及收集邮箱/分析）。
9. **埋点可能未铺全**（#26）：`track()` 封装在但调用点位需核实，否则 Plausible 上线也拿不到转化数据。
10. **Next.js 16 / Tailwind v4 较新**：`AGENTS.md` 明确警告 API 与训练数据不同；改代码前须查 `node_modules/next/dist/docs/`。Tailwind v4 无 config 文件，token 全在 `globals.css`，改样式需理解 4 层级联。

## 8. 给下游的建议（只点方向，详细架构留待 06-技术架构.md）

为把"已露出的功能"做实，预计需要引入的能力：

- **持久层（最关键）**：给候补名单/表单提交一个存储。轻量优先——
  - 邮件营销侧：**Resend Audiences** 或 Buttondown/Plunk（候补名单直接进可触达列表，解 #12）；
  - 结构化存档：Vercel Postgres / Neon / Supabase（若要后台看板/导出）。先邮件列表，后 DB。
- **可靠限流 / 反滥用**：**Upstash Redis + @upstash/ratelimit**（替换内存 Map），或 Vercel WAF / Cloudflare Turnstile 加人机校验。
- **内容管理**：把 blog/changelog/roadmap 从 TS 数组迁到 **MDX**（依赖已装，接线即可）或轻 CMS（Contentlayer/Velite/Keystatic），让运营改内容不改代码；blog 富文本/代码高亮复用现成 Shiki。
- **补齐"画饼"产物或调整文案**：要么真的产出 docs 站 / helm chart / binary release / Discord，要么按"诚实"原则把对应链接改成现状（否则违背品牌原则、损信任）。
- **CI/CD + 测试**：GitHub Actions 跑 `pnpm check` + `pnpm build` + `lint:links`/`lint:quickstart`；关键路径加 Playwright E2E（表单端到端、locale 切换、复制按钮、204/2xx 链接）。引擎侧也可接 qa-automation。
- **分析闭环**：核实并补全 `track()` 调用点位，确保 CTA 点击/表单提交/主题语言切换都打点，让 LAUNCH.md 的"48h 内至少一条转化"可观测。
- **法务**：补 privacy/terms 真实文本（GDPR/邮箱收集/Plausible 无 cookie 声明）。

## 摸底结论（给主管）

- **技术栈实况**：Next.js 16（App Router + `proxy.ts` 取代 middleware）+ React 19 + TS5 + Tailwind v4（无 config，OKLCH token 全在 globals.css）+ shadcn(radix-nova) + next-intl 4(EN根/ZH前缀) + Shiki SSR + framer-motion + RHF/zod；邮件 Resend、分析 Plausible，部署 Vercel（SSG+serverless）。**无 DB、无认证、无自有后端、无测试、无 CI。**
- **功能 shell vs 已实现占比**：约 28 个功能点 → **纯静态/占位 ≈39%、已完整(含前端交互/SEO/OG/i18n) ≈43%、部分依赖外部 env/账号 ≈18%**。**作为官网七成可用，作为产品功能几乎为空**（提交除发邮件外零沉淀）。
- **最大的几块硬骨头**：① 零持久层——尤其 **Cloud 候补名单只发邮件不入库**，无法后续触达；② 限流是内存 Map（serverless 下失效）；③ 一堆指向**尚不存在产物**的链接（docs站/helm repo/binary/Discord），照做会失败，违背"Proof over promise"；④ 内容全硬编码、MDX 半成品、无 CMS；⑤ 零自动化测试 + 零 CI。
