## 1. 指标体系

### 1.1 北极星指标（NSM）

> 北极星必须直接体现用户获得的价值，而非虚荣指标。

**NSM = 合格转化数**

```
合格转化数 = Cloud 候补提交数（真实入库可触达） + Design Partner 申请提交数（真实持久化存档）
```

**为什么是这个**：官网存在的核心商业目的是让"被 Datadog 账单刺痛的工程团队"进入 molesignal 的早期用户闭环。只有真正入库、创始团队能跟进的提交才算"合格"——仅发邮件不入库不计入（呼应 PRD P0-1/P0-2 的"不仅靠邮件"）。

**短期目标（上线后 48h）**：≥1 条真实转化事件可在 Plausible 观测（LAUNCH.md 要求）。

### 1.3 护栏指标

护栏指标监测优化过程中是否损害其他体验，避免"转化率优化"以牺牲信任为代价。

| 护栏指标 | 定义 | 阈值方向 | 观测手段 |
|---|---|---|---|
| **语言切换后离开率** | `locale_switch` 后 session 直接关闭的比例 | 不升（切换后内容体验不应变差） | Plausible + `locale_switch` 事件 |
| **深色/浅色主题切换率** | `theme_switch` 频率 | 不异常高（高频切换=对比度不适） | `theme_switch` 事件密度 |
| **表单错误后离开率** | 表单提交失败（400/429）后不再操作的比例 | 不升 | 无埋点；靠表单 conversion rate 倒推 |
| **A/B 实验流量纯洁度** | 对照/实验组不发生交叉污染 | 100% | session 维度分流一致性检查 |

## 2. 埋点方案（11 点位完整规格）

### 2.1 技术基础

- **SDK**：Plausible 浏览器脚本（`NEXT_PUBLIC_PLAUSIBLE_DOMAIN` 配置，仅 prod 加载）
- **调用方式**：`track(event, props?)` from `lib/analytics.ts`（no-op when `window.plausible` 不存在）
- **命名规范**：`snake_case` 动宾结构，全小写（`cta_click` ✅，`cta-click` ❌，`ctaClick` ❌）
- **隐私**：props **禁含任何 PII**（邮箱、姓名、IP 等），符合 Plausible 无 cookie 政策（PRD P0-6）
- **容错**：`track()` 已内置 try/catch，失败不冒泡，不阻断 UI

### 2.2 11 个事件完整规格

| # | 事件名 | 触发时机（精确） | props（字段 + 类型 + 枚举值） | 组件文件 | 去重/限制 |
|---|---|---|---|---|---|
| 1 | `cta_click` | 任意主/副 CTA 按钮或链接**被点击时** | `{ label: string, source_page: string, destination: string }` | Hero、Why、Start、Cloud、各页面 CTA 按钮 | 无（每次点击均发） |
| 2 | `demo_tab_switch` | CrossSignalDemo tab **实际切换时**（仅 tab 值变化才发） | `{ tab: "trace" \| "logs" \| "metric" }` | `components/cross-signal-demo.tsx` | 同 tab 重复点不发：在 `onValueChange` 内对比旧值 |
| 3 | `cost_calculator_interact` | CostCalculator 滑块**首次拖动**（每次组件挂载只发一次） | `{ ingest_gb: number, retention_days: number }` | `components/cost-calculator.tsx` | `useRef(false)` 守卫，首次拖动后置 `true`，后续静默 |
| 4 | `compare_table_expand` | "See full comparison" 展开按钮**点击时** | `{ source_page: string }` | `components/compare-table.tsx` | 无（每次点击均发） |

## 3. 转化漏斗

### 3.1 主漏斗（访客→转化）

```
[Step 1] 落地 /（首页）
         指标：PV + 唯一访客数（Plausible 默认）
         流失观测：跳出率（Plausible 默认）
             ↓ cta_click{destination="/why"}
[Step 2] /why 访问
         指标：PV、cost_calculator_interact 率
         流失观测：不交互计算器直接离开 → 兴趣不达预期
             ↓ cta_click{destination∈["/cloud","/design-partner"]} 或 cta_click{destination="/start"}
[Step 3] /start（技术考察）
         指标：PV、quickstart_copy 率、demo 探索深度（demo_tab_switch 次数）
         流失观测：复制后不再访问转化页 → 自托管意愿强但还未决策 Cloud/Partner
             ↓ cta_click{destination="/cloud"} 或 cta_click{destination="/design-partner"}
[Step 4] /cloud 或 /design-partner（转化页访问）
         指标：PV（Plausible 默认）
         流失观测：访问了转化页但不提交
             ↓ waitlist_submit 或 design_partner_submit
[Step 5] 表单提交成功（NSM 计入）
         指标：waitlist_submit 数、design_partner_submit 数
         → 合格转化数（NSM）
```

### 3.2 辅助漏斗（社区/开源兴趣）

```
首页 GitHub stats 可见 → github_star_click → 实际 star（仅在 GitHub 可观测，本站不可追踪完成）
changelog 访问 → rss_subscribe → 长期回访（RSS 阅读器端不可追踪）
```

### 3.3 漏斗观测关键流失点

| 流失节点 | 流失信号 | 干预方向（供后续迭代） |
|---|---|---|
| 首页 → Why 页跳出 | `cta_click{destination="/why"}` 少 / 首页跳出率高 | Hero 文案 A/B、滚动深度优化 |
| Why 页不交互计算器 | `cost_calculator_interact` / `/why` PV 比低 | 计算器可见性改善、初始值更吸引人 |
| Start 页复制后不转化 | `quickstart_copy` 高但转化页 PV 低 | Start 页 Cloud CTA 位置/文案 |
| 转化页访问不提交 | 转化页 PV 高但 `waitlist_submit`/`design_partner_submit` 少 | 表单长度、信任信号、社会证明 |

## 4. 留存框架

### 4.1 激活定义

**激活（Activation）**：用户完成至少一项有意图的交互行为：

| 优先级 | 激活动作 | 埋点信号 |
|---|---|---|
| **强激活**（高意图） | 提交 waitlist 或 Design Partner | `waitlist_submit` / `design_partner_submit` |
| **中激活**（探索意图） | 交互了 CostCalculator 或订阅 RSS | `cost_calculator_interact` / `rss_subscribe` |
| **弱激活**（兴趣意图） | 点击 GitHub star 链接 | `github_star_click` |

强激活用户是 NSM 的直接贡献者，是留存分析的核心群体。

### 4.2 留存分析框架

官网性质为营销落地站（非 SaaS 应用），留存的意义是"持续回访关注产品进展"：

| 留存维度 | 定义 | 可观测信号（Plausible） |
|---|---|---|
| **内容留存** | 周期性访问 changelog/blog/roadmap | 对应路径 PV 的日/周 cohort |
| **RSS 留存代理** | 订阅 RSS 后持续接收更新 | `rss_subscribe` 累计数（RSS 阅读端不可直接观测，以此为代理） |
| **意图留存** | 已激活用户（waitlist/partner）的后续回访 | Plausible 无用户 ID 概念，以 IP cohort 估算（仅内部参考，不作决策依据） |

**留存观测局限**：Plausible 无 cookie/无 uid，官网留存数据主要依赖运营邮件互动率（与 Resend 配合）而非前端埋点。

## 5. A/B 实验框架

### 5.1 分流方式

Plausible 本身不支持 A/B 分流，但可通过以下方式实现：

**方案（推荐）：服务端参数 + 自定义事件 props**

```typescript
// 服务端（Edge middleware 或 RSC）：按 session/cookie 分流，写入变体标识
// 客户端：在事件 props 中附加 variant 字段
track("cta_click", { label: "Get Started", source_page: "/", destination: "/why", variant: "A" });
```

- 分流规则：对访客 ID（Plausible 自生成，不含 PII）的末位数字取模，奇数 = A，偶数 = B
- Plausible Dashboard：过滤 `variant=A` / `variant=B` 对比目标事件转化率
- **无需第三方 A/B SDK**，满足"不引第二个分析 SDK"约束

### 5.2 首批值得做的实验假设

按预期影响力排序（NSM 优先）：

| 优先级 | 实验名 | 假设 | 实验变体 | 关键指标 | 判定方式 |
|---|---|---|---|---|---|
| **P0（首推）** | Hero CTA 文案 | "Star on GitHub" vs "Join the waitlist" 作为次级 CTA，影响候补转化路径 | A: 现有文案 / B: 更直接的候补 CTA | `waitlist_submit` 数（2 周） | 单尾 z-test，α=0.05，需至少 200 次首页访问 |
| **P1** | CostCalculator 初始值 | 初始 ingest=100GB/30 天 vs 更典型的生产值（500GB/90 天），后者预期省钱金额更大，更刺激转化 | A: 当前默认值 / B: 中高用量默认值 | `cost_calculator_interact` 率 + `cta_click{destination="/cloud"}` 转化率 | 同上 |
| **P2** | Start 页 Cloud CTA 位置 | 将 `/cloud` CTA 提到 QuickStart 完成步骤前展示（vs 现有底部） | A: 底部 / B: 步骤完成后插入 | `cta_click{destination="/cloud"}` 从 `/start` 的点击率 | 同上 |

### 5.3 实验纪律

- 最小样本量：每变体 ≥100 次目标事件（低流量站点，勿过度实验）
- 实验时长：至少 7 天，避免周末偏差
- 一次只跑一个实验（本站流量规模不支持 multi-variate）
- 护栏：实验期间监测整体转化率不下滑超过 10%，触发则终止
