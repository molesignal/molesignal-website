## 3. 数据 / 内容模型

### 3.1 Resend Contact 字段（候补 + 申请入库形态）

Resend Audiences 的 contact 原生字段有限（`email` / `firstName` / `lastName` / `unsubscribed`）。映射策略：

**Cloud 候补**（audience = `RESEND_CLOUD_AUDIENCE_ID`）：
```
{ email: <用户邮箱>, unsubscribed: false }
```
- 仅邮箱，进可触达列表即满足 P0-1。来源/时间戳通过通知邮件正文兜底留痕（Resend 不支持自定义字段时）。

**Design Partner 申请**（audience = `RESEND_PARTNER_AUDIENCE_ID`）：
```
{ email, firstName: <name 拆首词>, lastName: <name 余下>, unsubscribed: false }
```
> **name 拆 first/last 的确定性规则（T02/ISSUE-9 落地基准，QA/security 依此核对）**：对 `name.trim()` 先用 `\s+` 把内部连续空白塌缩为单个空格，再按**首个空格**切分——首段 = `firstName`，其余（保留中间单空格）= `lastName`；**单词姓名 → 仅 `firstName`，`lastName` 省略**（不传空串，靠 `addContact` 的 `firstName ? {...}` 条件拼接跳过）。例："Ada Lovelace"→first=Ada/last=Lovelace；"Ada"→仅 first=Ada；"Ada B. Lovelace"→first=Ada/last="B. Lovelace"；"  Ada   Lovelace  "→塌缩后同首例。映射逻辑**只在 route 内做**（建议抽 `lib/` 纯函数便于单测），不改 `addContact` 签名。
- 结构化字段（companySize / currentStack / biggestPain）**无法塞进 contact**，故 P0-2 的"持久化存档"= **进 partner audience（可触达）+ 通知邮件正文携带全部 5 字段**（创始团队在收件箱/Resend 可检索）。这是"不引 DB"约束下的最轻实现，满足 PRD"不仅靠邮件 → 至少进可触达 list + 字段在邮件正文可追溯"。
- 若后续需结构化看板（P1-5），运营在 **Resend Dashboard 导出 partner audience CSV**，字段从通知邮件归档补全；不在本期自建后台。

**幂等**：Resend `POST /audiences/{id}/contacts` 对已存在 email 返回成功（不报错），天然幂等（P0-1 验收④）。需在 provider 层显式处理 409/已存在为成功。

### 4.3 限流策略（统一）

```
provider: lib/rate-limit.ts 改造为 async，内部用 @upstash/ratelimit (滑动窗口)
  rateLimit({ key, max, windowMs }): Promise<{ ok, remaining, resetAt }>  // RateLimitResult 形状不变
  缺 UPSTASH_* env → fallback 到现有内存 Map 实现（同步逻辑包成 async）
阈值: cloud-waitlist 10/h · design-partner 5/h（保持现值）
超阈值: 429 + Retry-After 秒数 = Math.max(0, Math.ceil((resetAt-now)/1000))（非负整数）
key: `{form}:{ip}`，IP 取 x-forwarded-for 首段（getClientIp 已实现）
```

> **契约级行为约定（T11 落地基准）**：
> - **签名变更**：`rateLimit` 由 sync 改 async（红线②接口契约）；两个 route handler 调用处必须 `await`。`RateLimitResult` 字段（`ok/remaining/resetAt`）**不变**，故下游消费逻辑与 429 响应体不受影响。
> - **降级判定**：仅当 `UPSTASH_REDIS_REST_URL` 与 `UPSTASH_REDIS_REST_TOKEN` **同时存在**才启用 Upstash，否则走内存兜底；fallback warn **只打一次**（模块级 flag），不逐请求刷屏。
> - **fail-open 容错**：Upstash REST 运行时异常（网络/超时/限额）→ **放行请求**（返回 `ok:true`）+ warn，**绝不向用户返回 5xx**；表单可用性优先于限流严格性。
> - **实例缓存**：`Ratelimit` 实例按 `(max, windowMs)` 组合缓存复用（`Map` 键 `${max}:${windowMs}`），`Redis` 客户端模块单例，避免每请求重建；env 缺失时不得在 import 期实例化（防抛错）。
> - **运行时**：两 route 保持 Node serverless 默认运行时即可（Upstash REST 客户端 Node/Edge 通用），本期无需改 `export const runtime`。

### 4.6 Resend Audiences provider 接口（T01 新增 · T02 复用 · 并行对齐基准）

`lib/resend-audiences.ts`（T01/ISSUE-8 新建）。供 cloud 候补与 design-partner 申请**共用同一入库 provider**，是 T01↔T02 并行不撞车的契约面。

```ts
type AddContactResult = { ok: true } | { ok: false; skipped?: true; error?: string };

addContact(input: {
  audienceId: string | undefined;   // 调用方从 env 取（cloud/partner 各自的 AUDIENCE_ID）；空则跳过
  email: string;
  firstName?: string;               // partner 侧 name 拆首词；cloud 侧不传
  lastName?: string;                // partner 侧 name 余下；cloud 侧不传
  unsubscribed?: boolean;           // 默认 false
}): Promise<AddContactResult>
```

**行为契约（与 `lib/email.ts` 降级哲学一致，全程 fire-and-forget、绝不向上抛）**：
- **REST**：`POST https://api.resend.com/audiences/{audienceId}/contacts`，`Authorization: Bearer ${RESEND_API_KEY}`（复用 email.ts 同款 key，直 fetch，不引 SDK）。
- **幂等**：重复 email（Resend 返 409 / "already exists" 语义）→ **视为成功** `{ ok: true }`，不抛、不报错。
- **缺 `RESEND_API_KEY`** → `console.warn` + `{ ok: false, skipped: true }`（不发请求）。
- **缺 `audienceId`**（env 未配）→ 调用方判定跳过；provider 收到空 id **不对空 id 发请求**，返 `{ ok: false, skipped: true }` + warn（"audience id missing — skipping list insert"）。
- **运行时故障**（网络/5xx）→ log error 但 **不抛、不阻塞**，返 `{ ok: false, error }`；route handler 无论成败都返 200。
- **安全**：`audienceId`/`apiKey` **仅来自 env，不可被请求体覆盖**（无 SSRF/注入面）；错误日志不打印完整密钥（见 ISSUE-8 AC5）。

**调用方接线（route handler）**：zod 通过且非蜜罐后，**并行** fire-and-forget 触发 (a) `addContact(...)` 入对应 audience、(b) 既有 `sendEmail` founders 通知；二者皆不阻断 200。
> §4.1 处理顺序⑤⑥与 §3.1「幂等」由本契约统一落地；§4.1 文字描述为顺序，实现按本节**并行**触发（互不依赖、均 fire-and-forget）。
